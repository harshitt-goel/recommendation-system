# E-Commerce Recommendation System

Personalized product recommendations built on implicit user-item interaction data (views, add-to-carts, purchases), served via a REST API.

---

## Results

| Model | Recall@10 |
|---|---|
| Random baseline | 0.000257 |
| Popularity-based | — |
| **Matrix Factorization (ours)** | **0.006814** |

**~26x better than random** on a held-out leave-last-interaction-out test set (78,512 users, ~39K items, >99.9% sparse).

---

## Approach

1. **EDA** — event distribution, user/item activity, sparsity analysis
2. **Filtering** — removed users/items with fewer than 5 interactions (long-tail problem)
3. **Baseline 1** — popularity-based recommender (weighted: view=1, addtocart=5, transaction=10)
4. **Baseline 2** — item-based collaborative filtering (cosine similarity on top 2,000 items)
5. **Matrix Factorization** — learned 32-dim embeddings for every user and item using PyTorch
6. **Training** — implicit feedback via negative sampling + `BCEWithLogitsLoss` + L2 regularization (`weight_decay=1e-5`) to prevent overfitting
7. **Evaluation** — Recall@10 checkpointed every 5 epochs (leave-last-interaction-out split)
8. **Serving** — FastAPI REST API, Dockerized, with cold-start fallback to popularity ranking

## Quickstart

### 1. Train the model (in Jupyter)

Open `noteebooks/recommendation_system_clean.ipynb` and run all cells.
At the end, run the save cell:

```python
exec(open("../save_model.py").read())
```

This exports `models/model_weights.pt`, `models/model_config.pkl`, and `models/artifacts.pkl`.

### 2. Run API locally

```bash
pip install -r requirements.txt
uvicorn app.main:app --reload
```

Visit `http://localhost:8000/docs` for the interactive Swagger UI.

### 3. Run via Docker

```bash
# Build the image
docker build -t recsys-api .

# Run the container
docker run -p 8000:8000 recsys-api
```

## Key Design Decisions (Interview Notes)

- **Why BCEWithLogitsLoss, not MSELoss?** Labels are binary (0/1), not continuous ratings. BCE treats model output as a logit for "did user interact with this item?" — the correct framing for implicit feedback.
- **Why negative sampling?** All interactions in the data are positive. We generate random unseen items as negatives to train a binary classifier.
- **Why weight_decay (L2 regularization)?** Without it, the model drives training loss toward zero by memorizing training pairs — but Recall@10 on unseen test data stays near-random. Regularization keeps embeddings from overfitting.
- **Why leave-last-interaction-out split?** Random splits leak future behavior into training. Holding out each user's *most recent* interaction simulates the real deployment scenario.
- **Why popularity fallback for cold-start?** New users have no interaction history — the MF model has no embedding for them. Returning popular items is a simple, interpretable fallback that interviewers expect you to have thought about.
